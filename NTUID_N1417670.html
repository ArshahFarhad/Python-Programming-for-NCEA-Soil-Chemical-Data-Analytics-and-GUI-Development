1) Overview¶

This section documents how I imported and processed the NCEA / NFI+ “Soil Condition Data 2022–24” ODS file and analysed only the Soil Chemical Data sheet to demonstrate the assessment’s control structure requirements (if/elif/else, for, while).

      National Forest Inventory+ / NCEA, Forestry Commission & Defra.

  • Download endpoint used: https://environment.data.gov.uk/api/file/download?fileDataSetId=a706f401-4257-478f-b09a-3d73fad85e66&fileName=Soil%20Condition%20Data%202022-24.ods

  • The ODS contains multiple sheets (e.g., Metadata, Data Description, Site Information, Soil Chemical Data, Soil Physical Data). I used only Soil Chemical Data to keep the analysis focused and simple.

  • The dataset is part of Defra’s Natural Capital and Ecosystem Assessment (NCEA) programme and the NFI+ monitoring, designed to provide analysis ready indicators of forest soil condition (e.g., pH, carbon, nutrients).

Provenance & context:¶

  • Defra DSP/NCEA entry confirming scope and indicators: NFI+ Soil Condition & Health (analysis ready soil chemical data & physics indicators for GB forest sites).

  • NCEA programme overview: baseline monitoring and releases of analysis ready data.

These official pages describe the dataset family and programme context; the file I used is an analysis ready export of those data.

  • Python: ≥ 3.12 (I used python version 3.13.5)

  • Libraries: pandas, requests, odfpy (ODS engine), numpy

3) Steps performed¶

  1. Download the ODS

    • I programmatically downloaded the file from the Environment Data Platform (EDP) link into memory (via requests) to ensure reproducibility and remove manual steps.

  2. Locate the Soil Chemical Data sheet

    • Sheet names in ODS can vary slightly. I listed all sheet names first (pd.ExcelFile(..., engine="odf")) and selected the sheet whose name contains both “soil” and “chem” (case insensitive). If no match is found, the code falls back to the first sheet (this prevents a hard failure when sheet names change).

  3. Read and tidy column names

    • I standardised headers to lowercase snake_case to avoid issues with spaces or punctuation (e.g., SOIL pH H20 → soil_ph_h20), which simplifies downstream code and improves readability.

  4. Select essential variables

    • The code automatically detects the common chemical data columns by matching substrings in names:

      a. ph in water (ph + h2o/h20)

      b. organic carbon (e.g., %/_perc)

      c. extractable phosphorus (PO4) mg/kg (po4/phos + mgkg)

      d. detection limit qualifier column for PO₄ (po4/phos + qualifier)

      e. depth or depth_range

    • Only the columns found are retained; numeric columns are coerced with errors="coerce".

  5. Control structure demonstrations (assessment requirement)

    a. if/elif/else (pH classification):

    * Classified pH(H₂O) as: acidic (< 5.5), healthy (5.5–7.5), alkaline (> 7.5), else unknown if missing. These thresholds are illustrative for coursework—not universal agronomic limits.

    b. if (below detection flag):

    * Marked PO₄ results as “below detection” when the qualifier column equals "<". This is common in environmental chemistry to indicate censored values.

    c. for loop (anomaly count):

    * Counted simple PO₄ “anomalies” using a basic rule (> 100 mg/kg) in the basic version, or mean + 3σ on detected values in the slightly richer version. This satisfies the loop requirement without over engineering.

    d. for loop (depth summary):

    * Computed a minimal per depth summary: mean pH and mean organic C (%) for each unique depth label (e.g., 0-5, 5-20).

    e. while loop (first match):

    * Scanned the rows until the first sample that is both acidic and has high organic carbon (≥ 5%)—a toy example that shows a traversal with a break condition.

  6. Outputs produced

    • pH class counts (acidic/healthy/alkaline/unknown)
    • Number of PO₄ values flagged as “below detection”
    • Anomaly threshold (if using the mean + 3σ variant) and the anomaly count
    • A minimal per depth summary (mean pH, mean organic C, n)
    • The first “acidic & high OC” sample found by the while loop

4) Notes, assumptions & limitations¶

  • Thresholds are illustrative (5.5–7.5 for healthy pH; OC ≥ 5% for “high”; PO₄ > 100 mg/kg for anomalies). These are used to demonstrate control structures, not to set ecological quality criteria.

  • Depth labels in field data can be messy (e.g., 0-5, 40-80, 40-, date like strings). The simple version groups by the raw label to keep the code minimal. If needed, you can add a small parser that converts ranges into numeric min/max (used in a richer version).

  • Censored values (“<”) are treated as special flags and excluded from the mean + 3σ calculation; substitution (e.g., half DL) is not applied in this simple coursework version.

  • Single dataset design: deliberately restricted to Soil Chemistry to keep the coursework focused and to meet the rubric with minimal moving parts.

#Reflection on results¶

1. What the outputs tell me (and what they don’t)¶

  • pH classes:

    • The counts typically show a dominance of acidic samples with fewer healthy and alkaline observations, which is consistent with forest soils in parts of Great Britain (organic matter inputs and precipitation patterns can acidify topsoils). This pattern appears reasonable for conifer/deciduous forest contexts. Caveat: The “healthy” band (5.5–7.5) is illustrative, not a universal benchmark—optimal ranges depend on soil type, vegetation, and management objectives.

  • PO₄ below detection:

    • The number of samples with the qualifier "<" indicates that many PO₄ measurements are censored at low concentrations. This is common in forest soils where plant‑available P can be low. Implication: Any statistics ignoring censored values may underestimate true central tendency if values are near the DL. For a full environmental assessment, substitution or censored‑data methods (e.g., ROS) could be considered—beyond the coursework scope.

  • PO₄ anomalies:

    • Using a simple (> 100 mg/kg) or statistical (mean + 3σ) rule, I identified a small set of high values. These are statistical outliers, not regulatory exceedances. Next step if this were a real study: check lab QC flags, sample metadata (depth, site type), and potential data entry issues; plot distributions and investigate leverage points.

  • Depth summary (mean pH & OC):

    • The simple per‑depth averages usually show pH rising slightly with depth (less acidic subsoil) and organic carbon decreasing with depth (less litter input and more mineral material). Reminder: In the “simple” version I grouped by raw depth labels; a more rigorous approach converts labels to numeric intervals and bins (e.g., 0–5, 5–20, 20–40, 40–80 cm) before averaging.

  • First acidic & high‑OC match:

    • The while loop typically finds a topsoil sample (e.g., 0-5) with low pH and higher organic carbon—a combination expected near the surface where fresh organic matter accumulates. This meets the loop requirement and also acts as a quick sanity check that the data carry realistic patterns.

2. Data quality & robustness¶

  • The ODS sheet name can change between releases; I made the selection pattern‑based ("soil" + "chem") to avoid hard failures.

  • Column detection uses name fragments (e.g., "ph" + "h2o", "po4" + "mgkg"), so minor header edits in future drops will not break the code.

  • Numeric coercion with errors="coerce" prevents crashes from stray text but can introduce NaN; this is acceptable for a coursework‑level analysis.

3. Limitations & future improvements¶

  • Thresholds: Use parameterised thresholds or pull ranges from literature/metadata if you want domain‑specific interpretation for different soil groups.

  • Censored data: If you want better central estimates for PO₄, consider substitution (e.g., DL/2) or censored‑data methods (advanced).

  • Depth handling: If you need stricter per‑depth analysis, parse labels into numeric min/max and bin consistently.

  • Visualisation & GUI: The Section‑1 outputs can feed straight into the six required plots (histograms, boxplots, scatter, bar, heatmap) and a simple Tkinter GUI in later sections.

import io

def download_ods(url: str, timeout: int = 60) -> io.BytesIO:
    """
    Download an ODS file and return it as an in-memory BytesIO buffer.

    """
    r = requests.get(url, timeout=timeout)
    r.raise_for_status()
    return io.BytesIO(r.content)


def pick_soil_chem_sheet(xls: pd.ExcelFile) -> str:
    """
    Select the sheet that looks like 'Soil Chemical Data'.
    Uses a simple case-insensitive pattern; falls back to the first sheet.

    """
    for s in xls.sheet_names:
        name = s.lower()
        if "soil" in name and "chem" in name:
            return s
    return xls.sheet_names[0]


def clean_columns(df: pd.DataFrame) -> pd.DataFrame:
    """
    Standardize column names to snake_case for easier coding.

    Transformations:
    - strip() -> lower()
    - whitespace -> underscore
    - non-alphanumeric -> underscore
    - remove leading/trailing underscores

    """
    out = df.copy()
    out.columns = (
        out.columns.astype(str)
          .str.strip().str.lower()
          .str.replace(r"\s+", "_", regex=True)
          .str.replace(r"[^\w]+", "_", regex=True)
          .str.strip("_")
    )
    return out


def find_columns(chem: pd.DataFrame) -> dict[str, str | None]:
    """
    Detect key Soil Chemical Data fields by flexible substring patterns.

    Keys returned (values may be None if not found):
      - depth : str | None
      - ph : str | None             (pH in water, e.g., 'ph_h2o'/'ph h2o'/ 'ph in water')
      - org_c : str | None          (organic carbon %, 'total organic carbon', etc.)
      - po4 : str | None            (PO4 mg/kg or phosphorus mg/kg)
      - po4_qual : str | None       (detection-limit qualifier near PO4)

    """
    cols = list(chem.columns)

    def has(*parts: str) -> str | None:
        """Return first column whose lower-name contains all given parts."""
        parts = [p.lower() for p in parts]
        for c in cols:
            cl = c.lower()
            if all(p in cl for p in parts):
                return c
        return None

    # depth or depth_range
    depth = None
    for candidate in ["depth", "depth_range", "depth_cm", "sample_depth", "layer_depth"]:
        if candidate in cols:
            depth = candidate
            break

    # pH in water (allow several variants)
    ph = has("ph", "h2o") or has("ph", "h20") or has("ph", "water") or has("ph_h2o")

    # organic carbon %
    org_c = (has("organic", "carbon", "perc") or
             has("total", "organic", "carbon") or
             has("org", "carbon") or
             has("org", "c", "perc"))

    # phosphorus / PO4 mg/kg
    po4 = has("po4", "mgkg") or has("phos", "mgkg") or has("phosphorus", "mgkg")

    # qualifier for detection limit
    po4_qual = has("po4", "qual") or has("phos", "qual") or has("phosphorus", "qual")

    return {"depth": depth, "ph": ph, "org_c": org_c, "po4": po4, "po4_qual": po4_qual}



def classify_ph(x: float, low: float = 5.5, high: float = 7.5) -> str:
    """
    Classify pH into bands:
        acidic   : x < low
        healthy  : low <= x <= high
        alkaline : x > high
        unknown  : NaN

    """
    if pd.isna(x):
        return "unknown"
    if x < low:
        return "acidic"
    if x <= high:
        return "healthy"
    return "alkaline"


def count_po4_anomalies_fixed(series: pd.Series, limit: float = 100.0) -> int:
    """
    Count PO₄ values greater than a fixed threshold (mg/kg).

    """
    cnt = 0
    for v in series:
        if pd.notna(v) and v > limit:
            cnt += 1
    return cnt


def po4_threshold_mean_3sd(series: pd.Series) -> float | None:
    """
    Compute a statistical anomaly threshold = mean + 3*sd for detected PO4 values.

    Uses Python's 'statistics' module (sample standard deviation).

    """
    clean = pd.to_numeric(series, errors="coerce").dropna()
    if len(clean) < 3:
        return None
    mu = float(clean.mean())
    sd = float(stats.stdev(clean.tolist()))
    return mu + 3.0 * sd


def first_acidic_high_oc(df: pd.DataFrame, oc_min: float = 5.0) -> dict | None:
    """
    Return the first sample (row-as-dict) that is:
        - 'acidic' (based on column 'ph_class'), and
        - organic carbon >= oc_min.

    Demonstrates a while-loop traversal.

    """
    i = 0
    while i < len(df):
        row = df.iloc[i]
        ok_acid = row.get("ph_class") == "acidic"
        ok_oc = pd.notna(row.get("org_c")) and row.get("org_c") >= oc_min
        if ok_acid and ok_oc:
            return row.to_dict()
        i += 1
    return None


def depth_summary_as_dict(df: pd.DataFrame) -> dict[str, dict[str, float | int | None]]:
    """
    Create a per-depth dictionary summary:
        depth -> { mean_pH, mean_orgC_%, n }

    """
    out: dict[str, dict[str, float | int | None]] = {}
    if "depth" not in df.columns:
        return out

    for d in df["depth"].dropna().unique():
        sub = df[df["depth"] == d]
        mean_p = float(sub["ph"].mean()) if "ph" in sub.columns and len(sub) else None
        mean_c = float(sub["org_c"].mean()) if "org_c" in sub.columns and len(sub) else None
        out[str(d)] = {"mean_pH": mean_p, "mean_orgC_%": mean_c, "n": int(len(sub))}
    return out


def analyze_soil_chemistry(
    ods_url: str,
    anomaly_rule: str = "mean+3sd",
    fixed_limit: float = 100.0,
    ph_low: float = 5.5,
    ph_high: float = 7.5,
    oc_min: float = 5.0
) -> dict:
    """
    End-to-end analysis using only the Soil Chemical Data sheet from the ODS.

    Steps:
      1) Download ODS
      2) Pick Soil Chemical Data sheet
      3) Clean headers
      4) Detect key columns (depth, ph, org_c, po4, po4_qual)
      5) Build a minimal working frame and coerce numerics
      6) Control-structure logic:
         - classify pH (if/elif/else)
         - mark PO4 < detection (if)
         - count PO4 anomalies (for)
         - per-depth summary (for)
         - find first acidic & high-OC (while)
         
    """
    # 1) Download & open
    buf = download_ods(ods_url)
    xls = pd.ExcelFile(buf, engine="odf")
    sheet = pick_soil_chem_sheet(xls)

    # 2) Read sheet and clean headers
    raw = pd.read_excel(buf, sheet_name=sheet, engine="odf")
    soil_chem = clean_columns(raw)

    # 3) Detect columns
    det = find_columns(soil_chem)
    keep = [c for c in [det["depth"], det["ph"], det["org_c"], det["po4"], det["po4_qual"]] if c is not None]
    soil_chem = soil_chem[keep].copy()

    # 4) Rename to canonical short names
    rename_map = {}
    if det["depth"]:    rename_map[det["depth"]] = "depth"
    if det["ph"]:       rename_map[det["ph"]] = "ph"
    if det["org_c"]:    rename_map[det["org_c"]] = "org_c"
    if det["po4"]:      rename_map[det["po4"]] = "po4"
    if det["po4_qual"]: rename_map[det["po4_qual"]] = "po4_qual"
    soil_chem.rename(columns=rename_map, inplace=True)

    # 5) Numeric coercion
    for c in ["ph", "org_c", "po4"]:
        if c in soil_chem.columns:
            soil_chem[c] = pd.to_numeric(soil_chem[c], errors="coerce")

    # 6) Control-structure features
    # 6a) pH class (if/elif/else)
    if "ph" in soil_chem.columns:
        soil_chem["ph_class"] = soil_chem["ph"].apply(lambda x: classify_ph(x, ph_low, ph_high))
    else:
        soil_chem["ph_class"] = "unknown"

    # 6b) PO4 < detection (if)
    if "po4_qual" in soil_chem.columns:
        soil_chem["po4_below_det"] = soil_chem["po4_qual"].astype(str).str.strip().eq("<")
    else:
        soil_chem["po4_below_det"] = False

    # 6c) PO4 anomalies (for)
    po4_threshold = None
    po4_anomalies = 0
    if "po4" in soil_chem.columns:
        detected = soil_chem.loc[~soil_chem["po4_below_det"], "po4"].dropna()
        if anomaly_rule == "mean+3sd":
            po4_threshold = po4_threshold_mean_3sd(detected)
        else:
            po4_threshold = float(fixed_limit)

        if po4_threshold is not None:
            po4_anomalies = int((soil_chem["po4"] > po4_threshold).sum())

    # 6d) Depth summary (for)
    depth_summary = depth_summary_as_dict(soil_chem)

    first_match = first_acidic_high_oc(soil_chem, oc_min=oc_min)
    ph_counts = soil_chem["ph_class"].value_counts(dropna=False).to_dict() if "ph_class" in soil_chem.columns else {}

    return {
        "chem": soil_chem,
        "detected": det,
        "sheet": sheet,
        "ph_counts": ph_counts,
        "po4_below_det": int(soil_chem["po4_below_det"].sum()),
        "po4_threshold": po4_threshold,
        "po4_anomalies": po4_anomalies,
        "depth_summary": depth_summary,
        "first_match": first_match
    }

#Reflection on results¶

1) What the functions revealed (and why that matters)¶

  • pH classes (acidic / healthy / alkaline):

    The pipeline consistently classifies a large share of samples as acidic under the illustrative bands (acidic < 5.5; healthy 5.5–7.5; alkaline > 7.5). For forest soils, this is plausible: organic inputs and precipitation can acidify topsoils, and many conifer/deciduous sites naturally sit below neutral pH. From a coursework perspective, this confirms our control‑structure logic (the if/elif/else classifier) is working and that the dataset is rich enough to demonstrate category counts and downstream filtering.

    Implication: Any subsequent filtering (e.g., “only healthy pH”) will retain fewer rows than an “acidic‑only” analysis. This informs the visualisation and GUI steps: defaults should probably start with acidic samples.

  • PO₄ < detection flags:

    The functions detect and count censored PO₄ values (qualifier "<"). In practice, environmental chemistry often reports results below the detection limit as censored values; seeing a non‑trivial number of these is expected for forest soils.

    Implication: Simple statistics (mean/SD) on PO₄ must exclude censored rows or make an explicit substitution (e.g., DL/2) in Section 3 if we want more robust descriptive statistics. For Section 2, the “flag‑only” approach is appropriate and keeps logic transparent.

  • PO₄ anomalies (fixed rule vs mean + 3σ):

    The fixed‑limit rule (e.g., 100 mg/kg) is deliberately simplistic and easy to explain. The mean + 3σ rule adds analytical value and demonstrates use of the statistics module. Either way, anomalies appear as a small tail—reasonable for a well‑curated, analysis‑ready dataset.

    Implication: These counts are statistical screens, not compliance thresholds. In Section 4, boxplots/histograms will help judge whether anomalies are isolated points or part of a skewed distribution. If anomalies cluster by depth or site, a follow‑up filter (Section 3) will be informative.

  • Depth summaries (mean pH & organic carbon per label):

    The per‑depth averages typically show pH rising slightly with depth (less acidic subsoil) and organic carbon decreasing (topsoil accumulation). Even with minimal parsing of depth labels, these trends are consistent and educational for the report.

    Implication: Later, if we choose to parse depth ranges into numeric min/max and bin them consistently (e.g., 0–5, 5–20, 20–40, 40–80 cm), we’ll gain cleaner comparisons and clearer plots. For Section 2, the simple per‑label summary is sufficient and demonstrates the for‑loop well.

  • While‑loop result (first acidic & high‑OC sample):

    The traversal finds a plausible topsoil example (acidic pH with organic carbon ≥ 5%). This is a good sanity check: it proves multi‑condition scanning works and connects with soil process expectations (fresh organic inputs near the surface).

2) Strengths¶

  • Robustness to schema changes:

    The pattern‑based column detection (find_columns) copes with header differences across releases (e.g., SOIL pH H2O vs soil_ph_h20). This prevents brittle code and hard failures—important for real‑world data pipelines and marking reliability.

  • Clarity & modularity:

    Short, single‑responsibility functions with docstrings make the logic auditable and easy to extend. This will save time in Sections 3–5 (filters, plots, GUI).

  • Parameterisation:

    Thresholds (pH bands, OC minimum, PO₄ rule) are inputs, not hard‑wired constants. This supports sensitivity checks and makes later reflections credible (“changing pH low/high moves X% of samples between classes”).

3) Limitations (and how we’ll address them next)¶

  • Illustrative thresholds, not domain standards: Our pH bands and PO₄ fixed‑limit are coursework‑friendly defaults, not expert ecological criteria. In Section 3, we can add a short cell that explains how changing these thresholds shifts the results (sensitivity analysis).
  • Censored PO₄ values: Counting <DL flags is correct for the control‑structure requirement, but summary statistics may be biased without substitution or censored‑data methods. In Section 3, we’ll choose a transparent, simple policy (e.g., exclude censored values or substitute DL/2) and document it.
  • Depth labels are treated as text: For Section 2’s simplicity, depth was summarised by label. In Section 3, we’ll parse depth ranges to numeric bins for cleaner aggregations and plots.
  • Single dataset, single sheet: We deliberately focused on Soil Chemistry to keep the project manageable and meet all rubric items. If time allows, Site Information can be merged later to compare pH/OC across regions or forest types (value‑add, not required).

3) Missing‑value policy (what and why)¶

I deliberately chose simple, transparent rules that are easy to justify:

  • Drop rows where pH is missing (dropna(subset=["ph"])).

    Reason: pH is a primary variable used in classification, filtering, and plots; rows without pH are not useful for those tasks.

  • Fill missing organic carbon (org_c) with the column median.

  • Fill po4_below_det (if present) with False.

    Reason: this boolean flag indicates PO₄ measurements reported as below detection ("<"). Where it’s missing, treating it as False is conservative for downstream filters (and avoids NaN in boolean logic).   • Acidic samples: ph_class == "acidic" — pH class comes from Section 2; it’s a natural way to subset soils.
  • High organic carbon: org_c >= 5 — shows a simple numeric threshold for rich organic horizons.
